# Бекенд обліку електроенергії

REST API на **FastAPI** з **PostgreSQL** (SQLAlchemy 2.x), міграції **Alembic**. Центральний сервіс платформи: реєстр об’єктів мережі, прийом сирих показів, валідація, агреговані звіти, порогові сповіщення, топологія та інтеграція з **Prometheus** (`/metrics`).

## Навіщо він у стенді

- Приймає покази з симулятора або реальних джерел (`POST /readings`).
- Зберігає й обробляє дані в єдиній моделі **enterprise → site → grid → meter**.
- Дає фронтенду та скриптам стабільний контракт (`/docs` — OpenAPI).

## Потік даних (спрощено)

1. **Raw readings** — таблиця сирих показів після `POST /readings`.
2. **Validation** — синхронізація в **validated_readings** з базовими прапорцями якості (розриви, спайки тощо).
3. **Reports** — добові/місячні агрегати; **`POST /reports/rebuild`** перераховує зведення з validated.
4. **Alerts** — правила порогів за сумою споживання у вікні; **`POST /alerts/run`** (або автоматично після показу) створює записи сповіщень.

## Модулі API (роутери)

| Префікс / зона | Файл | Призначення |
|----------------|------|-------------|
| `/health`, `/ready` | `app/routers/system.py` | Живість процесу та перевірка БД. |
| `/sites`, `/meters`, `/enterprises`, … | `registry.py` | Реєстр об’єктів і лічильників. |
| `/readings` | `readings.py` | Прийом показів; після збереження — sync validation + оцінка порогів. |
| `/validation` | `validation.py` | Керування/запуск синхронізації validated. |
| `/reports`, `/reports/...` | `reports.py`, `reports_analytics.py` | Агрегати та аналітичні зрізи. |
| `/alerts` | `alerts.py` | Правила, список сповіщень, resolve, ручний run. |
| `/topology`, мережеві ендпоінти | `topology.py`, `network.py` | Топологія мережі та огляд зв’язків. |
| `/metrics` | `app/metrics.py` | Метрики Prometheus. |

Повний список маршрутів — у **`http://<host>:8000/docs`**.

## Конфігурація (ENV)

Базові змінні через **`app/core/config.py`** (також підтримується `.env`):

| Змінна | Призначення |
|--------|-------------|
| **`DATABASE_URL`** | Рядок підключення SQLAlchemy (наприклад `postgresql+psycopg2://…`). |
| **`CORS_ORIGINS`** | Список origin через кому для браузерного фронту та dev-серверів. |
| **`ALERT_DEFAULT_THRESHOLD_KWH`** | Дефолтний поріг для логіки за замовчуванням (якщо використовується). |

## Міграції та дані

- Каталог **`alembic/`** — версії схеми; у compose окремий сервіс **`backend-migrate`** виконує `alembic upgrade head`.
- Початкове наповнення — скрипти в **`scripts/`** (наприклад `seed_data.py`) з кореня репозиторію.

## Тести

Інтеграційні pytest-тести (HTTP до запущеного API): **`tests/`**, конфіг **`pytest.ini`**. Локально або в Docker — див. коментарі в `tests/test_api_pipeline_integration.py`. Залежності для розробки — **`requirements-dev.txt`**; у production-образ також може бути встановлено **`pytest`** для запуску всередині контейнера.

## Зв’язок із деплоєм

У **`deploy/compose/docker-compose.yml`** задаються **`DATABASE_URL`**, **`CORS_ORIGINS`**, порт **8000**. Для Swarm — секрети та обгортки в **`deploy/swarm/`** і **`deploy/swarm/SECRETS.md`**.

Коротко: **бекенд — це єдиний шлюз даних і бізнес-логіки обліку**, на який орієнтуються фронтенд, симулятор і моніторинг.
