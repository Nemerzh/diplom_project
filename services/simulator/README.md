# Симулятор показів лічильників

Окремий сервіс (**simulator v2**), який імітує потік реальних показів енергії й надсилає їх у бекенд через **`POST /readings`**. Він не зберігає довгострокову історію сам по собі: джерело правди — API обліку (`services/backend`), а симулятор лише генерує **кВт·год за інтервал** для кожного активного лічильника та публікує їх у систему.

## Навіщо він у стенді

- Наповнює **raw readings → validation → aggregates → alerts** без фізичних лічильників.
- Дає **кероване навантаження**: інтервал тиків, добові профілі, «погані» події (offline, spike) для перевірки якості та моніторингу.
- Експонує **метрики Prometheus** (успішні покази, ретраї, помилки POST).

## Як це працює

1. **Список лічильників** періодично знімається з бекенду: `GET {API_URL}/meters`. Беруться лише записи зі статусом **`active`** (за потреби обмеження через **`METER_IDS`** у форматі `1,2,5`).
2. У нескінченному циклі (asyncio) для кожного лічильника обчислюється **`value_kwh`** за поточний інтервал часу на основі типу/ролі/зони, години доби, глобального профілю та опцій з **`scenario.yaml`**.
3. Згенерований JSON надсилається на **`POST {API_URL}/readings`** з полями `meter_id`, `ts`, `value_kwh`, `source: "simulator"`.
4. Якщо бекенд тимчасово недоступний, **`poster.py`** повторює запит з **експоненційним backoff** (налаштовується в конфігу).

Шлях до API задається змінною **`API_URL`** (у Docker Compose зазвичай `http://backend:8000`).

## Конфігурація: `scenario.yaml` і ENV

Файл **`scenario.yaml`** (або шлях через **`SIM_CONFIG_PATH`**) зливається з **вбудованими дефолтами** у `config_loader.py`. Змінні середовища **перекривають** окремі поля без редагування YAML (наприклад **`INTERVAL_SECONDS`**, **`SIM_PROFILE`**, **`SIM_MULTIPLIER`** — зворотна сумісність із compose).

Основні блоки:

| Блок / поле | Призначення |
|-------------|-------------|
| **`defaults.interval_seconds`** | Період між тиками циклу (з легким **jitter** за `jitter_fraction`). |
| **`defaults.meters_refresh_seconds`** | Як часто оновлювати список лічильників з API. |
| **`defaults.global_profile`** | Режим навантаження: `normal` \| `peak` \| `critical` \| `offline` (масштабує або вимикає генерацію). |
| **`defaults.multiplier`** | Глобальний множник інтенсивності. |
| **`defaults.smoothing_alpha`** | Згладжування «потужності» між тиками (0 — більше шуму, 1 — швидша реакція на шаблон). |
| **`defaults.post`** | Ретраї POST: `max_retries`, `backoff_seconds`, `backoff_multiplier`. |
| **`events`** | Ймовірність **spike**, множники spike, ймовірність і тривалість **offline** (пропуск показів на випадковий інтервал). |
| **`meter_overrides`** | Перевизначення для конкретного **ID лічильника**: наприклад `multiplier`, `smoothing_alpha`, `emit_every_n_ticks` (рідші відправки). |

Генератор (`generator.py`) враховує **тип лічильника** (`electricity`, `water`, `gas`, `heat`) і **роль/зону** (основний, освітлення, HVAC, виробництво тощо) для діапазонів потужності та добового шаблону.

## Керування симулятором (HTTP API самого сервісу)

| Метод | Шлях | Дія |
|-------|------|-----|
| `POST` | `/simulator/start` | Запуск циклу симуляції (якщо ще не запущено). |
| `POST` | `/simulator/stop` | Зупинка циклу. |
| `POST` | `/simulator/reload-config` | Перечитати `scenario.yaml` з диска. |
| `GET` | `/simulator/status` | Чи працює цикл, профіль, інтервал, кількість відстежуваних станів. |

Якщо встановлено **`SIM_AUTOSTART=1`**, цикл стартує при піднятті процесу без окремого виклику `/simulator/start`.

Додатково: **`/health`**, **`/ready`**, **`/metrics`** (Prometheus).

## Зв’язок із деплоєм

У **`deploy/compose/docker-compose.yml`** симулятор збирається з `services/simulator`, отримує **`API_URL`**, опційно монтується **`scenario.yaml`**, задаються **`SIM_CONFIG_PATH`** та змінні для автостарту й інтервалу. Це дозволяє змінювати сценарій на хості без перезбірки образу (якщо змонтовано файл).

Коротко: **симулятор — це генератор синтетичних показів із конфігурованою поведінкою та стійкою доставкою в бекенд**, щоб увесь пайплайн обліку й моніторинг працювали на демо-стенді без реального обладнання.
